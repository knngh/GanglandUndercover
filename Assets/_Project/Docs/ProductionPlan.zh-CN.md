# 港区潜线：联网版完整开发 Plan

## 1. 产品定位

《港区潜线》是一款 Unity 制作的联网社交推理游戏。核心体验接近“多人在线身份推理 + 场景任务 + 尸体报告 + 会议投票”，但题材和系统是自己的香港警匪卧底故事。

目标不是文字原型，也不是单机 Demo，而是一个 6-10 人在线房间游戏：

- 玩家进入同一张香港港区地图。
- 系统暗中分配警察、卧底、黑帮等身份。
- 警方阵营通过任务、证据链、监控和目击锁定黑帮。
- 黑帮阵营通过暗杀、破坏、伪证、断电和反侦察干扰调查。
- 卧底要在黑帮和警方之间走钢丝，传递证据但不能暴露。
- 死亡或紧急按钮触发会议，玩家用语音/文字争辩并投票。
- 每局 8-12 分钟，快速开房、快速重开、强重玩性。

## 2. 当前工程定位

当前 Unity 工程已经从离线玩法验证推进到“联网优先可玩骨架”：

- 主场景启动后优先进入在线港区原型。
- 已接入 Netcode for GameObjects 与 Unity Transport。
- 已有 Host / Client / Ready / Start / Leave 的基础房间流。
- Host 可通过本机 IP 进行局域网或本机多开测试。
- 玩家人数不足时会自动补 AI，使单个 Unity 实例也能跑完整一局。
- 已有隐藏身份、任务同步、黑帮破坏、击倒、尸体、报案、紧急会议、投票、跳票、出局、胜负结算和案情日志。

原离线玩法验证场仍保留为参考和回归测试资产，但新开发应优先强化在线对局质量。

从现在开始，所有新增系统都要按联网需求设计：

- 玩家输入只提交意图，不直接改最终状态。
- 房主/服务器统一判定击倒、任务进度、会议、投票和胜负。
- 身份、私有线索、黑帮队友信息必须按玩家私有权限下发。
- 任务、尸体、门、电闸、摄像头、会议状态必须可网络同步。
- UI 要区分本地玩家视角和公共对局状态。

## 2.1 当前可玩版本状态

当前版本已经具备一局完整游戏的最小闭环：

1. 创建 Host。
2. 开始在线局，系统自动补齐 AI 到最少 5 名参与者。
3. 系统分配警方、卧底、黑帮身份。
4. 玩家和 AI 在港区地图移动。
5. 警方/卧底推进证据任务。
6. 黑帮破坏任务、触发黑灯、击倒目标。
7. 尸体或紧急铃触发会议。
8. 玩家和 AI 投票或跳票。
9. 出局后公开身份，继续行动或结算。
10. 警方通过证据链或清除黑帮胜利；黑帮通过人数压制胜利。

当前版本仍不是最终商业发行质量，也不能按成熟游戏评价。它目前更准确的定位是“联网玩法骨架 + 粗糙视觉占位”：一局流程可以跑通，但美术语言、角色动画、任务表现、会议质感、音频和地图构图都还离目标很远。下一阶段不能继续靠运行时方块和临时 UI 硬堆内容，必须转向正式垂直切片。

## 2.2 当前差距判断

当前最主要差距：

1. 场景不是作者化地图。现有地图主要靠程序化生成和临时模型堆叠，缺少统一构图、区域节奏、光照层次、导航线和可识别房间入口。
2. 美术不是成品管线。角色仍是占位胶囊/简化模型，缺少统一角色风格、骨骼动画、身份皮肤、死亡状态和会议入座表现。
3. 任务只是玩法占位。已有任务状态、进度和面板，但还不是正式小游戏，也缺少每个任务的专属动画、音效和失败/破坏反馈。
4. UI 仍是工程化 HUD。主菜单、房间、行动、案情板、大地图、会议、投票、结算还没有达到商业产品的完整视觉层级。
5. 音频还是占位。运行时提示音能验证事件，但缺少正式音乐、环境声、交互音效、会议提示和紧张度变化。
6. 联网服务还没生产化。Cloud/Auth/Lobby/Relay 基础可用，但 Vivox 当前仍显示未配置，外部多人测试、断线重连、举报/静音/封禁都还没完成。
7. 部分免费素材导入质量不稳定。LowpolyStreetPack 的 `.meta` GUID 无效导致部分模型被 Unity 忽略，不能把正式表现建立在这批损坏导入上。

因此接下来的目标不是继续宣称“接近发行版”，而是先做一张可展示、可试玩、可扩展的高质量垂直切片。垂直切片通过后，再扩成完整大地图。

## 3. 推荐技术栈

### Unity 版本

继续使用当前本机已打开的 Unity `6000.4.5f1`，避免频繁升级项目。

### 联机方案

第一阶段采用：

- Netcode for GameObjects：适合当前 GameObject / MonoBehaviour 项目、小规模实时同步对局。
- Unity Transport：底层传输。
- Lobby：房间列表、邀请码、私密房。
- Relay：先做低成本房主托管联机，方便外网测试，不先上专用服务器。
- Authentication：匿名登录起步，后续接账号。
- Vivox：语音/文字会议、存活频道、死亡频道、黑帮频道。
- Multiplayer Play Mode：本机多客户端测试。

第二阶段再评估：

- Dedicated Server / Multiplay Hosting：如果需要反作弊、排行榜、正式运营。
- Matchmaker：如果玩家量起来后需要自动匹配。

依据：

- Unity 官方多人文档把 Lobby、Relay、Matchmaker、Netcode 列为多人连接和实时同步的核心组件。
- Unity 官方说明 Netcode for GameObjects 适合 GameObject / MonoBehaviour 工作流和较小规模实时同步。
- Unity 官方 Vivox 文档支持跨平台语音、文字和安全能力。

## 4. 核心在线玩法

### 4.1 标准对局

人数：6-10 人。

推荐配置：

- 6 人：1 黑帮、1 卧底、4 警方/市民侧。
- 8 人：2 黑帮、1 卧底、5 警方/市民侧。
- 10 人：2 黑帮、1 黑帮辅助、1 卧底、6 警方/市民侧。

单局流程：

1. Lobby 等待：房主设置人数、地图、语音、公开/私密房。
2. 准备阶段：所有玩家 Ready。
3. 身份分配：服务器/房主随机分配，私发身份。
4. 行动阶段：玩家自由移动，完成任务、破坏、监控、接头、暗杀。
5. 报告阶段：发现尸体或按紧急按钮，所有人传送到会议。
6. 会议阶段：展示有限证据，开启语音/文字讨论。
7. 投票阶段：公开投票结果，但不一定公开身份，按房间规则决定。
8. 继续行动或结算。

### 4.2 胜负条件

警方阵营胜利：

- 完成证据链并触发收网。
- 投出所有黑帮核心成员。

黑帮阵营胜利：

- 黑帮人数 >= 非黑帮存活人数。
- 倒计时结束且证据链未完成。
- 卧底被识破并清除，且警方证据链断裂。

卧底个人胜利加成：

- 警方胜利时卧底仍存活。
- 卧底暴露值低于阈值。
- 至少传递过一条关键证据。

## 5. 身份设计

### 警方阵营

反黑督察：

- 能在会议中查看一条增强证据。
- 每局有限次数“重点盘问”。

技侦警员：

- 可修复摄像头、电闸和门禁。
- 可调取短时间路线片段。

鉴证员：

- 处理尸体更快。
- 能区分强线索/弱线索。

冲锋队员：

- 可短时间封锁一条路。
- 可触发短追捕，成功后获得强嫌疑线索。

### 卧底阵营

潜伏探员：

- 可进入黑帮区域。
- 可秘密上传证据。
- 暴露值过高会被黑帮会议或技能识破。

线人：

- 可获得黑帮路线片段。
- 证词可信度波动，适合制造推理不确定性。

### 黑帮阵营

打手：

- 近距离击倒目标。
- 击倒后会留下脚印、路线或目击风险。

白纸扇：

- 伪造证词。
- 篡改监控。
- 降低黑帮热度但制造假线索。

车手：

- 可使用快速路线或后巷撤离。
- 适合制造不在场证明。

坐馆代理：

- 能触发一次大型破坏事件。
- 能看到黑帮队友状态。

## 6. 场景重新设计

### 首张在线地图：九龙港区夜局

在线地图不能只是摆房间，要支持追逐、目击、绕路、断电和会议推理。

核心区域：

- 货柜码头：开阔、有长视线，适合追踪和转移证物。
- 夜市街：NPC/遮挡多，适合伪装和目击冲突。
- 茶餐厅：中立接头点，可做证词任务。
- 后巷：短路多，黑帮逃脱和暗杀高发。
- 天台：风险路线，可观察部分街区。
- 地下诊所：黑帮藏匿、伪造伤情、处理痕迹。
- 临时指挥车：警方证据上传核心点。
- 警署外勤站：紧急会议、身份确认、封锁授权。
- 电房：断电/修复核心点。
- 监控室：技侦任务核心点。

地图结构要求：

- 至少 10 个命名区域。
- 至少 3 条主路、5 条短路、2 条高风险逃生路线。
- 每个区域至少关联一个任务或事件。
- 每个区域能产生至少一种线索：目击、门禁、脚印、摄像头、证词、物证。
- 每个关键点至少有两条可达路径，避免单点堵死。

## 7. 联网任务模型

任务不做纯文字按钮，必须是短交互小游戏，并且要能网络同步。

### 警方任务

调取监控：

- 本地小游戏：在短片段中选择可疑身影。
- 同步结果：证据链增加、某区域生成路线线索。

查封货柜：

- 本地小游戏：比对封条、货单、箱号。
- 同步结果：货柜状态变为已查封。

修复电闸：

- 本地小游戏：连接正确线路。
- 同步结果：断电结束、摄像头恢复。

扫描证物：

- 本地小游戏：指纹/血迹/弹壳比对。
- 同步结果：生成强线索或弱线索。

无线电监听：

- 本地小游戏：锁频、过滤杂音。
- 同步结果：短时间显示黑帮活动区域。

上传档案：

- 本地小游戏：校验文件顺序。
- 同步结果：证据链进度增加。

### 黑帮任务

篡改监控：

- 关闭某摄像头一段时间。
- 生成假路线或删除片段。

制造断电：

- 降低视野，关闭监控。
- 给黑帮提供逃脱窗口。

伪造证词：

- 在会议中污染证据。
- 提高某个无辜玩家嫌疑。

转移证物：

- 把已发现证据移动到其他区域。
- 降低证据链完整度。

威吓证人：

- 让某条证词延迟出现或可信度下降。

### 卧底任务

秘密接头：

- 在茶餐厅、后巷、天台等点位传证。
- 增加证据链，同时增加暴露值。

双面任务：

- 表面做黑帮任务，实际留下警方线索。
- 失败会大幅增加暴露值。

## 8. 网络同步规则

### 必须同步的公共状态

- 玩家位置、朝向、移动状态。
- 玩家存活/倒地/被投出局状态。
- 任务点状态、任务进度、破坏状态。
- 断电、电闸、摄像头、门禁、封锁状态。
- 尸体位置、报告状态。
- 会议阶段、投票阶段、倒计时。
- 公开证据、公开路线、会议结果。

### 私有状态

- 身份。
- 黑帮队友列表。
- 卧底暴露值。
- 私有任务目标。
- 私有线索。
- 技侦增强线索。

### 权威模型

MVP 阶段：

- Host authoritative。
- 客户端只发送输入意图：移动、交互、击倒、报告、投票。
- Host 判定合法性后同步结果。

正式测试阶段：

- 对关键动作增加服务器验证。
- 击倒、报告、投票、任务完成、身份分配必须由权威端决定。
- 客户端不能直接写胜负、身份、证据链。

## 9. 在线系统清单

Lobby：

- 创建公开房。
- 创建私密房。
- 加入码。
- 房间列表。
- 玩家 Ready。
- 房主踢人。
- 房间设置：人数、地图、语音、任务数量、匿名投票。

Session / Relay：

- 创建在线会话。
- 加入会话。
- 断线重连第一版。
- 房间结束后返回 Lobby。

语音/文字：

- 行动阶段：默认近距离语音，可设置关闭。
- 会议阶段：全员语音。
- 黑帮频道：可选，平衡后决定是否开放。
- 死亡频道：死亡玩家可交流但不能影响存活玩家。
- 文字快捷发言：适配无麦玩家。

匹配：

- MVP 先做房间码。
- Alpha 做公开房列表。
- Beta 做快速匹配。

## 11. 美术与表现

方向：

- 2.5D 香港雨夜警匪片。
- 霓虹、雨水反光、警灯、码头雾气。
- 角色不是太空服，而是便衣警察、黑帮打手、卧底、技侦、夜市人员。

角色：

- 每个身份有剪影差异，但不能一眼暴露阵营。
- 皮肤不影响判定体积。

- 监控台、电闸箱、货柜封条、证物柜、无线电台、档案终端。
- 每个任务点要有明确视觉状态：可用、进行中、已完成、被破坏。

动作：

- 移动。
- 交互。
- 击倒。
- 报告。
- 修复。
- 破坏。
- 会议入座。
- 被投出局。

## 12. 开发里程碑

### M0.5：垂直切片重做

目标：

- 先做一块 2-3 分钟可试玩的高质量样板区，而不是继续扩粗糙大地图。
- 样板区必须同时证明美术、角色、任务、HUD、会议和联网状态能达到目标方向。

样板区范围：

- 中央会议点。
- 监控室。
- 茶餐厅。
- 夜市通道。
- 后巷/电房入口。
- 4 个任务点：监控调取、线人录音、电闸修复、车牌追踪。
- 6-8 个玩家出生/会议座位点。

交付：

- 一张可编辑 prefab 场景，不再只靠运行时方块生成。
- 统一低多边形/2.5D 港区美术包，损坏或缺 GUID 的素材不作为主资源。
- 一套临时但统一的角色 prefab：便衣、警员、黑帮、卧底、路人。
- 基础 Animator：移动、交互、击倒、死亡、报案、会议入座。
- 4 个正式任务 UI 小游戏，不再是纯数字按钮。
- 会议 UI 第一版：头像、发言计时、证据卡、投票按钮、结果揭示。
- Vivox 状态在 UI 中明确显示：大厅、行动近距离、会议频道。
- 第一屏截图必须可读：玩家、任务、房间、行动条同时清楚出现。

验收：

- 打开 Play Online Demo 后第一屏不再像方块原型。
- 玩家无需说明能辨认自己、任务点、房间入口和当前操作。
- 4 个任务至少 3 个有专属交互动画/音效。
- 会议和投票界面达到可给外部玩家试玩的完整程度。
- 烟测继续通过，并新增垂直切片场景、任务 UI、会议 UI、角色 prefab 的检查。

### M0：当前工程收口

目标：

- 保证当前 Unity 工程可编译、可 Play、可跑烟测。
- 当前 Demo 作为离线验证场，不继续当最终方向。

交付：

- 黑屏修复保留。
- 当前 Demo 能跑。
- 文档切换到联网版计划。

### M1：联网最小闭环

目标：

- 两到四个本机客户端可进入同一房间。
- 玩家能同步移动和看到彼此。

交付：

- NetworkManager。
- Player NetworkObject。
- Host/Client UI。
- Relay/Session 或本地直连测试。
- Multiplayer Play Mode 验证。

验收：

- 2 个客户端能进入同一场景。
- 移动、出生、断开、重连基础流程可用。

## 13. 正式发行必须申请的 API / 服务

### 13.1 当前 Unity 路线的必需项

这些是“像太空狼人杀一样能公网联机、开房、开会、投票、语音、发行”的最低服务组合：

1. Unity Cloud Project / Unity Dashboard 项目：所有 Unity Gaming Services 的项目容器，必须绑定当前 Unity 工程。
2. Unity Authentication：玩家身份、匿名登录、Steam / Apple / Google Play 等平台账号绑定。必须有它，Lobby、Relay、Vivox、Analytics 等服务才能稳定识别同一玩家。
3. Unity Lobby：房间列表、房间码、公开/私密房、Ready 状态、房间规则、重连入口。
4. Unity Relay：不先做专用服务器时，用于外网 Host 托管、NAT 穿透和跨网络联机。当前本地 IP 直连不能支撑商业发行。
5. Vivox Voice & Text：行动阶段近距离语音、会议全员语音、死亡频道、黑帮频道、文字聊天、静音/屏蔽/举报证据。
6. Analytics：记录开局、退房、任务完成、击倒、会议、投票、胜负、留存、崩溃前行为，用于平衡和商业运营。
7. Diagnostics / Crash & Exception Reporting 或 Sentry 等替代：崩溃、异常、设备、版本、网络错误日志。正式测试和上线必须有。按 2026-05-03 的 Unity 文档，旧 Cloud Diagnostics 已提示未来会被 Unity Dashboard 新诊断体验替代，因此后续接入时优先看当前 Unity Dashboard 的 Diagnostics 入口。
8. 举报、封禁、聊天安全和客服后台：可以先用 Vivox Safe Text / Moderation SDK 或自建后台，但有语音/文字的联网游戏必须有玩家安全处理链路。

### 13.2 发行平台必需项

Steam 版：

1. Steamworks 开发者账号、AppID、商店页、构建上传权限。
2. Steam 登录 / 票据校验：验证 Steam ID 和游戏拥有权。
3. Steam 好友邀请、Overlay、成就、云存档、排行榜按需求接入。
4. 如果只在 Steam 发行，可以评估 Steam Lobby / Networking；如果未来跨平台，仍建议以 Unity Lobby / Relay / Dedicated Server 为主。

iOS / macOS 版：

1. Apple Developer Program。
2. App Store Connect 应用、Bundle ID、证书、描述文件、TestFlight。
3. Game Center 登录、成就、排行榜或多人能力。
4. StoreKit / Apple In-App Purchase：只有做皮肤、通行证、DLC、订阅时必需。
5. 隐私清单、数据收集说明、麦克风权限、用户生成内容/聊天审核说明。

Android / Google Play 版：

1. Google Play Console 开发者账号、应用包名、签名、测试轨道和生产发布权限。
2. Google Play Games Services：Android 玩家平台登录、成就、排行榜、云存档按需求接入。
3. Google Play Billing：只有做皮肤、通行证、DLC、订阅或消耗品时必需。
4. 数据安全表单、麦克风权限说明、目标 SDK / 内容分级 / 广告 ID 声明。

### 13.3 是否需要地图 API

如果继续当前“虚构港区 + 自制程序化/手工场景”路线：不需要地图 API，这是最稳的商用路线。

如果要接入真实香港 3D 地图底座：

1. 香港 CSDI / Lands Department 3D Visualisation Map API：官方 3D Tiles 数据源，需申请 API key、遵守使用条款、署名、并注意并发和带宽限制。
2. Cesium for Unity / Cesium ion token：用于在 Unity 中加载和管理 3D Tiles，生产环境要创建独立 token，不用默认开发 token。
3. OSM / Overture Maps 数据管线：可作为道路、建筑轮廓和 POI 的替代底图来源，但仍要做授权、缓存、离线化和美术改造。

按 2026-05-03 查询，香港 CSDI 的 3D Spatial Data API 已在 2026-03-16 更新 tileset URL 结构，旧结构将于 2026-05-09 后不再支持。如果未来接真实 3D 港区地图，必须按当时官方文档使用最新地址和 key 规则。

商业发行建议：首发不要依赖实时地图 API。先用原创九龙港区地图发行，真实地图只作为后续资料参考或特殊地图 DLC，避免授权、性能、网络和审核风险拖慢主线。

### 13.4 可选但强烈建议的后台

1. 自建轻量后端：账号绑定、封禁名单、举报处理、赛季配置、公告、热更新参数。
2. Dedicated Server / 云服务器：正式反作弊、排行榜、竞技匹配、跨平台稳定性需要。MVP 可以 Relay，商业长期不应只靠玩家 Host。
3. Matchmaker：玩家量起来后做快速匹配、地区匹配、段位匹配；首发小规模测试可先不用。
4. Remote Config / Game Overrides：不用发新包就调整证据目标、黑帮人数、任务数、冷却时间、活动规则。
5. Economy / IAP 后台：只有做外观、通行证、赛季商店时启用。不要在核心玩法未稳前先做复杂经济。
